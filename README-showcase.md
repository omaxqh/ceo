# CEO Skill

商业智能外脑。从创始人视角审视你的想法，发现风险、激发可能、推动决策。

---

## 这个 Skill 做什么

**核心价值：** 让你的商业想法在 ship 之前经历一次严苛的 CEO 级审查——商业结构、竞争格局、用户洞察、战略轨迹、体验设计，全覆盖。

**解决什么问题：**
- 创始人/PM 独自思考时的盲点和自我确认偏误
- 商业计划缺乏结构化质疑和压力测试
- 在 scope 膨胀和过度精简之间找不到平衡

**适用场景：**

| 场景 | 你会得到什么 |
|------|-------------|
| 新创业方向验证 | 前提质疑、方案对比、竞对分析、3 个月理想状态映射 |
| 现有产品新功能规划 | scope 边界厘定、扩展机会筛选、风险登记表 |
| PRD/提案审核 | 逻辑漏洞检测、边缘案例覆盖、跨模型第二意见 |

---

## 四种审查模式

| 模式 | 中文释义 | 适用场景 | 典型触发语 |
|------|----------|----------|------------|
| **SCOPE EXPANSION** | 梦想大一点 | 全新商业想法/创业方向 | "go big"、"ambitious" |
| **SELECTIVE EXPANSION** | 保持范围 + 挑选扩展 | 现有业务的新功能/新方向 | "show me options" |
| **HOLD SCOPE** | 最大严谨 | 验证一个具体假设 | "就这个范围"、"别扩展" |
| **SCOPE REDUCTION** | 精简到本质 | 主动聚焦/砍方向 | "MVP"、"最小版本" |

**关键规则：** 用户 100% 掌控。每个 scope 变更都需要明确确认——不会悄悄添加或删除任何内容。

## CEO 思维框架

18 条思维模式，来自 Bezos、Munger、Jobs、Grove、Horowitz、Hastings、Graham、Altman、Rams 等思想家。这不是检查清单，而是内化到审查过程中的思维本能。

### 决策与行动

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 1 | 分类本能 | 按可逆性 × 影响大小分类决策。大多数事是双向门，快速行动 |
| 6 | 速度校准 | 快是默认。70% 信息就足够决定 |
| 12 | 勇气积累 | 信心来自做出困难决定，不是在之前 |
| 13 | 意志力即战略 | 有意识地固执。大多数人放弃得太早 |

### 风险与偏执

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 2 | 偏执扫描 | 持续扫描战略拐点、文化漂移、人才流失、指标代理病 |
| 3 | 逆向反思 | 对每个「怎么赢」也问「什么会让我们失败」 |
| 7 | 代理指标怀疑 | 指标还在服务用户，还是已经自我参照？ |

### 聚焦与减法

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 4 | 聚焦即减法 | 主要价值在于不做什么。Jobs 把 350 个产品砍到 10 个 |
| 14 | 杠杆执念 | 找到小投入产生大产出的输入点 |
| 17 | 减法默认 | 尽可能少的设计。功能膨胀比缺少功能更快杀死产品 |

### 人与组织

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 5 | 人优先排序 | 人、产品、利润——永远按这个顺序 |
| 10 | 创始人模式 | 深度参与不是微管理，如果它扩展（而非限制）团队思考 |
| 11 | 战时意识 | 正确诊断和平 vs 战时。和平习惯会杀死战时公司 |

### 长期与架构

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 8 | 叙事连贯 | 困难决定需要清晰框架。让「为什么」可读 |
| 9 | 时间深度 | 用 5-10 年视角思考。对重大赌注应用后悔最小化 |

### 体验与信任

| 编号 | 框架 | 核心洞察 |
|------|------|----------|
| 15 | 层级即服务 | 用户首先、其次、第三看到什么？尊重时间 |
| 16 | 边缘案例偏执 | 空状态是功能，不是事后补救 |
| 18 | 信任设计 | 每个界面决策要么建立要么侵蚀用户信任 |

## 审查工作流

```
PRE-REVIEW
    │
    ├─ 背景文档读取
    ├─ UI/体验范围检测
    └─ Landscape Check（市场竞对调研）
           │
           ▼
Step 0: 核心质疑 + 模式选择 ← 用户显式选择
    │
    ├─ 0A. 前提质疑：这是正确的问题吗？
    ├─ 0B. 现有方案利用：市场上已有什么？
    ├─ 0C. 理想状态映射：3 个月后这应该是什么样？
    ├─ 0C-bis. 方案对比（强制）：至少 2-3 个实现路径
    ├─ 0D. 模式特定分析
    ├─ 0E. 执行时序审查
    └─ 0F. 模式选择确认
           │
           ▼
Review Sections（深度审查）
    │
    ├─ Section 1:  商业结构审查
    ├─ Section 10: 战略轨迹审查
    ├─ ──── CHECKPOINT（战略层/执行层分界）────
    ├─ Section 4:  用户行为边缘案例
    └─ Section 11: 设计与体验审查
           │
           ▼
第二意见（Outside Voice）
    │
    ├─ 派遣独立子 agent 进行残酷诚实审查
    └─ 跨模型紧张处理 ← 用户决定采纳哪个意见
           │
           ▼
Required Outputs（强制输出）
    │
    └─ 自动落盘到 reviews/<project-slug>/report.md
```

**核心特性：**
- **显式用户掌控：** 每个 scope 变更、每个模式切换都需要用户确认
- **跨模型验证：** 第二意见由独立子 agent 提供，捕捉主审查的盲点
- **战略层/执行层分离：** 可在 CHECKPOINT 处结束，适合只验证方向的场景

## 输出物

审查完成后你会得到：

### 分析产出

| 输出 | 内容 |
|------|------|
| 风险登记表 | 场景、风险、可能性、影响、缓解措施、可逆性 |
| 行动清单 | 需验证的商业假设 + 需进一步调研的方向 |
| NOT in scope | 考虑过但明确延迟的方向，附理由 |
| 已有方案分析 | 市场上已有哪些可利用的产品/工具/平台 |
| 3 个月理想状态差距 | 这个计划让你离目标还有多远 |

### 图表产出

| 图表 | 展示内容 |
|------|----------|
| 商业模式图 | 价值主张、收入来源、关键资源 |
| 竞争格局图 | 市场定位、差异化维度 |
| 用户旅程图 | 关键触点、情感曲线 |
| 3 个月里程碑图 | 关键节点、决策依赖 |

### 决策记录

| 记录 | 作用 |
|------|------|
| 决策记录表 | 每个 AskUserQuestion 的问答历史 |
| Scope 变更历史 | 初始 scope → 变更过程 → 最终 scope |

## 安装

### 通过 ali-skills CLI（推荐）

```bash
# 安装到当前项目
npx ali-skills add <group>/ceo-skill

# 安装到指定 Agent
npx ali-skills add <group>/ceo-skill --skill ceo -a qoder
npx ali-skills add <group>/ceo-skill --skill ceo -a claude-code
npx ali-skills add <group>/ceo-skill --skill ceo -a cursor

# 全局安装
npx ali-skills add <group>/ceo-skill --skill ceo -g
```

### IDE Skill（手动）

| 平台 | 安装方式 |
|------|----------|
| Qoder | 复制目录到 `~/.qoder/skills/ceo/` |
| Cursor | 复制目录到 `.cursor/skills/ceo/` |
| Codex | 整个目录放入 skills 目录 |
| Claude Code | 贴到 `CLAUDE.md` 或 `.claude/commands/` |

### LLM 直接使用

把 `how-great-ceos-think.md` 丢到 GPT/Gemini/Claude 对话框：

```
使用这个 md 作为核心思考和方法论
```

然后发送需要审核的方案。

## 核心原则

这些原则贯穿整个审查过程：

```
零静默失败        每个失败模式都必须可见
失败场景优先      资金断裂？用户不付费？核心供应商撤退？
边缘案例偏执      首次用户、误操作、网络中断、状态过期
图表是必须的      没有重要流程可以不画图
延迟的事情要写下来  模糊的意图是谎言
为 3 个月后优化    不只是解决今天的问题
你有权说推倒重来   如果有根本更好的方向，提出来
```
